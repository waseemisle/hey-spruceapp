Implementation Guide

Stripe Recurring & Auto-Debit

GroundOps (Hey Spruce) — Planning document for automatic payment deduction

Version: 1.0 Date: March 2, 2025

This document explains how to implement two billing scenarios using Stripe in the GroundOps application:

  • Fixed recurring: Automatic deduction of a fixed amount (e.g., $300) on a specific day each month (e.g., 2nd of every month).
  • Variable recurring: First-time charge when the customer provides card details; subsequent months auto-charge the invoice amount using the stored card.

1. Current Application State

GroundOps is a facility maintenance and work order platform built with:

  • Stack: Next.js 14, TypeScript, Firebase (Firestore, Auth, Storage), Stripe, Mailgun
  • Collections: clients, invoices, scheduled_invoices, recurringWorkOrders, recurringWorkOrderExecutions
  • Current Stripe flow: One-time Stripe Checkout sessions for invoices. Each invoice generates a payment link; customer pays manually; webhook marks invoice as paid. No saved cards or recurring logic.

Existing Stripe API routes:

  • /api/stripe/create-payment-link — Creates Checkout session for an invoice (one-time payment).
  • /api/stripe/webhook — Handles checkout.session.completed, checkout.session.expired, payment_intent.payment_failed.

Neither the Client nor Invoice types store Stripe customer IDs or payment methods. Cards are not saved today.

2. Scenario 1: Fixed Amount on Fixed Date

Example

Customer A pays $300 on the 2nd of every month automatically from their card/account.

Concept

Stripe supports this via Subscriptions with a billing cycle anchor. You create a monthly subscription with:

  • Fixed price: $300/month
  • Billing anchor: day of month = 2 (so billing happens on the 2nd)

Stripe will automatically charge the customer on the 2nd of each month without any manual action.

3. Scenario 2: Variable Amount Each Month

Example

Customer B:

  • Month 1: Invoice is $400 → Customer enters card once → $400 is charged immediately.
  • Month 2: Invoice is $500 → System uses stored card → $500 is charged automatically (no customer action).
  • Month 3+: Same pattern: invoice amount varies; charge uses saved card.

Concept

This is not a subscription (amounts change). It requires:

  1. Setup phase: Use Stripe Setup Intents to collect and save the card (no initial charge).
  2. Storage: Attach the payment method to a Stripe Customer; store stripeCustomerId and (optionally) default paymentMethodId on your Client record.
  3. Charging phase: When an invoice is due, create a Payment Intent with the invoice amount and charge the saved PaymentMethod. This is an “off-session” charge.

Stripe supports this workflow; it is commonly used by utility companies and similar businesses that charge variable amounts each cycle.

4. Does Stripe Support This?

Yes Stripe supports both scenarios.

Scenario Stripe Product Key APIs / Features
Fixed amount on fixed date Subscriptions Subscriptions API, billing_cycle_anchor, Products & Prices
Variable amount each month Setup Intents + Payment Intents Setup Intents (save card), Payment Intents (charge saved card off-session)

Important: For off-session charging (Case 2), you must obtain explicit customer consent and comply with Strong Customer Authentication (SCA). Stripe’s Setup Intents handle SCA when collecting the card.

5. Implementation Overview

5.1 Scenario 1: Fixed Recurring

Database

  • Store stripeCustomerId on the Client document (or a separate billing subcollection).
  • Store stripeSubscriptionId and link it to the scheduled/fixed plan.

New / Updated API Routes

  • Create Stripe Customer: On client onboarding or when enabling auto-pay, create a Stripe Customer and save the ID.
  • Create Subscription: Create a Product (e.g., “Monthly Facility Maintenance”) with a Price of $300/month. Create a Subscription with billing_cycle_anchor set to day 2.
  • Checkout for Subscription: Use Stripe Checkout in subscription mode, or use Elements + Subscription API.

Flow (high level)

1. Admin or client opts in to “Auto-pay $300 on 2nd of month”
2. App creates or fetches Stripe Customer, creates Product/Price for $300/month
3. Customer completes Stripe Checkout (subscription mode)
4. Stripe starts subscription with billing anchor on day 2
5. Stripe auto-charges on the 2nd each month
6. Webhooks (invoice.paid, invoice.payment_failed) update your app

Integration Points

Align with scheduled_invoices or recurringWorkOrders: when a client has a fixed monthly amount and day, create a Stripe Subscription instead of (or in addition to) sending a payment link.

5.2 Scenario 2: Variable Recurring

Database

  • Store stripeCustomerId on Client.
  • Store defaultPaymentMethodId for the saved card.
  • Invoices: add fields like autoChargeAttempted, autoChargeStatus.

New / Updated API Routes

  • Create Stripe Customer: When client opts into auto-pay.
  • Setup Intent: Endpoint that creates a SetupIntent; frontend uses Elements or Checkout (setup mode) to collect the card.
  • Charge Saved Card: Backend endpoint (or cron) that creates a PaymentIntent with off_session: true.
  • Webhooks: Handle payment_intent.succeeded, payment_intent.payment_failed.

Flow (high level)

1. Client receives first invoice ($400) and clicks “Pay” or “Save card for auto-pay”
2. App creates SetupIntent; client enters card
3. PaymentMethod attached to Customer; stored in DB
4. First invoice: charge immediately via PaymentIntent
5. Future invoices: cron finds due invoices for clients with saved cards
6. Create PaymentIntent with amount = invoice total, off_session: true
7. Webhook updates invoice status; send receipt email

Integration Points

  • Invoice creation: When invoice is created for a client with saved card, charge immediately or queue for due-date charge.
  • Recurring work order execution: Use charge-saved-card flow instead of one-time payment link.
  • Scheduled invoices: Same charge logic for variable amounts.

5.3 Firestore Security Rules

Ensure only admins (or authenticated backend) can write stripeCustomerId and defaultPaymentMethodId. Consider a clientPaymentMethods subcollection for multiple cards per client.

6. Challenges & Considerations

6.1 Technical Challenges

Challenge Mitigation
Off-session charges can fail Retry logic; notify client; fall back to manual payment link
Card updates or re-auth Setup Intent for card updates; handle payment_intent.requires_action
Time zones and charge timing Store client timezone; schedule cron appropriately
Idempotency Use idempotency keys when creating PaymentIntents

6.2 Compliance & Legal

Mandatory consent for off-session charging

You must clearly obtain consent: (1) how the amount is determined, (2) payment frequency, (3) authorization to charge the saved card. Use a checkbox and terms during the save-card flow.

PCI compliance

Never store raw card numbers. Stripe handles card data; you store only Stripe Customer and PaymentMethod IDs.

6.3 User Experience

  • “Manage payment method” or “Update card” flow in the client portal
  • Advance notice before auto-charging (e.g., email 3–5 days before)
  • Receipts immediately after successful charge
  • Clear instructions when charges fail

7. Recommended Phased Approach

1

Phase 1 — Save card (no auto-charge)

Add Setup Intent flow; save Stripe Customer + PaymentMethod for clients. Extend Client schema. No automatic charging yet.

2

Phase 2 — Variable auto-charge

Implement charge-saved-card endpoint; add cron; integrate with invoice due dates. Handle webhooks and retries.

3

Phase 3 — Fixed recurring

Implement Stripe Subscriptions for clients with fixed monthly amounts; integrate with scheduled_invoices.

4

Phase 4 — Polish

Client portal UI for managing cards, email notifications, retry logic, reporting.
